# HTTP와 서버-브라우저 통신 구조 정리

## 1. HTTP란?

HTTP는 **HyperText Transfer Protocol**의 약자입니다.

쉽게 말하면, **브라우저와 서버가 웹에서 데이터를 주고받기 위한 약속**입니다.

예를 들어 사용자가 크롬에서 웹사이트 주소를 입력하면 브라우저는 서버에게 요청을 보냅니다.

```text
브라우저 → 서버: 이 페이지 주세요
서버 → 브라우저: 여기 HTML/CSS/이미지/데이터입니다
```

이때 요청과 응답을 어떤 형식으로 주고받을지 정해둔 규칙이 바로 **HTTP**입니다.

---

## 2. 브라우저란?

브라우저는 사용자가 웹사이트를 볼 수 있게 해주는 프로그램입니다.

대표적인 브라우저는 다음과 같습니다.

```text
Chrome
Safari
Edge
Firefox
```

브라우저의 역할은 다음과 같습니다.

```text
1. 사용자가 입력한 주소를 해석한다.
2. 서버에 HTTP 요청을 보낸다.
3. 서버가 보내준 HTML, CSS, JavaScript, 이미지 등을 받는다.
4. 받은 파일들을 해석해서 화면에 보여준다.
```

즉, 브라우저는 **요청을 보내는 쪽**입니다.

그래서 브라우저를 보통 **클라이언트 Client**라고 부릅니다.

---

## 3. 서버란?

서버는 사용자의 요청을 기다리고 있다가, 요청이 오면 응답을 보내주는 프로그램입니다.

서버는 단순히 큰 컴퓨터만 의미하지 않습니다.

정확히는 다음과 같은 역할을 하는 **프로그램**입니다.

```text
요청을 기다린다.
요청이 오면 읽는다.
요청에 맞는 응답을 만든다.
응답을 클라이언트에게 보낸다.
```

예를 들어 42 `webserv` 과제에서 직접 만드는 프로그램도 서버입니다.

```bash
./webserv
```

이 프로그램이 실행되면 특정 IP와 포트에서 브라우저의 요청을 기다립니다.

```text
127.0.0.1:8080
0.0.0.0:8080
example.com:80
example.com:443
```

여기서 `8080`, `80`, `443` 같은 숫자를 **포트 번호**라고 합니다.

---

## 4. 서버와 브라우저는 어떻게 연결되는가?

서버와 브라우저는 처음부터 연결되어 있는 것이 아닙니다.

사용자가 브라우저에 주소를 입력하면, 브라우저가 먼저 서버를 찾아가면서 연결이 만들어집니다.

전체 흐름은 다음과 같습니다.

```text
사용자
  ↓ 주소 입력
브라우저
  ↓ 서버 주소 찾기
DNS
  ↓ IP 주소 반환
브라우저
  ↓ 서버에 연결 요청
서버
  ↓ 응답 전송
브라우저
  ↓ 화면 출력
사용자
```

---

## 5. 사용자가 주소를 입력하면 생기는 일

예를 들어 사용자가 브라우저에 다음 주소를 입력했다고 가정합니다.

```text
https://example.com/index.html
```

브라우저는 먼저 `example.com`이라는 도메인이 실제로 어떤 컴퓨터를 가리키는지 알아야 합니다.

사람은 `example.com` 같은 도메인 이름을 사용하지만, 컴퓨터는 IP 주소로 통신합니다.

```text
example.com → 93.184.216.34
```

이 변환을 해주는 시스템이 **DNS**입니다.

```text
브라우저: example.com의 IP 주소가 뭐야?
DNS: 93.184.216.34야
```

---

## 6. IP와 포트

IP 주소는 서버 컴퓨터를 찾기 위한 주소입니다.

포트는 그 컴퓨터 안에서 어떤 프로그램과 통신할지 정하는 번호입니다.

예를 들어 다음과 같은 주소가 있다고 합시다.

```text
93.184.216.34:443
```

의미는 다음과 같습니다.

```text
93.184.216.34 → 서버 컴퓨터의 주소
443            → HTTPS 서버 프로그램이 기다리는 포트
```

HTTP와 HTTPS의 기본 포트는 다음과 같습니다.

```text
HTTP  → 80번 포트
HTTPS → 443번 포트
```

만약 직접 서버를 실행할 때 `8080` 포트를 사용한다면 브라우저에서 이렇게 접속합니다.

```text
http://localhost:8080
```

---

## 7. TCP 연결

HTTP 요청을 보내기 전에 보통 먼저 **TCP 연결**이 만들어집니다.

TCP는 데이터를 안정적으로 주고받기 위한 통신 방식입니다.

쉽게 표현하면 다음과 같습니다.

```text
브라우저: 연결해도 돼?
서버: 응, 연결하자.
브라우저: 알겠어.
```

이 과정을 TCP 3-way handshake라고 부릅니다.

```text
브라우저 → 서버: SYN
서버 → 브라우저: SYN + ACK
브라우저 → 서버: ACK
```

이 연결이 만들어진 후에야 브라우저와 서버는 데이터를 주고받을 수 있습니다.

---

## 8. HTTPS라면 TLS 과정이 추가됨

주소가 `https://`로 시작하면 TCP 연결 이후에 **TLS 암호화 과정**이 추가됩니다.

```text
브라우저: 너 진짜 example.com 서버 맞아?
서버: 인증서 보여줄게.
브라우저: 확인했어. 이제 암호화해서 통신하자.
```

이 과정 덕분에 로그인 정보, 비밀번호, 결제 정보 같은 중요한 데이터가 중간에서 쉽게 노출되지 않습니다.

정리하면 다음과 같습니다.

```text
HTTP  = 암호화 없는 기본 웹 통신
HTTPS = HTTP + TLS 암호화
```

---

## 9. HTTP 요청 Request

연결이 만들어지면 브라우저는 서버에게 HTTP 요청을 보냅니다.

예시는 다음과 같습니다.

```http
GET /index.html HTTP/1.1
Host: example.com
```

이 요청의 의미는 다음과 같습니다.

```text
example.com 서버야,
index.html 파일을 보내줘.
```

여기서 `GET`은 데이터를 요청할 때 사용하는 HTTP 메서드입니다.

---

## 10. HTTP 응답 Response

서버는 브라우저의 요청을 읽고 응답을 보냅니다.

예시는 다음과 같습니다.

```http
HTTP/1.1 200 OK
Content-Type: text/html

<html>
  <h1>Hello</h1>
</html>
```

이 응답의 의미는 다음과 같습니다.

```text
요청이 성공했고,
HTML 문서를 보내줄게.
```

브라우저는 이 응답을 받아서 HTML을 해석하고 화면에 출력합니다.

---

## 11. 브라우저가 화면을 그리는 과정

서버가 HTML 파일 하나만 보내면 끝나는 것이 아닙니다.

HTML 안에는 CSS, JavaScript, 이미지 파일이 연결되어 있을 수 있습니다.

예를 들어 HTML 안에 다음 코드가 있다고 합시다.

```html
<link rel="stylesheet" href="/style.css">
<script src="/app.js"></script>
<img src="/logo.png">
```

그러면 브라우저는 추가로 서버에 요청을 보냅니다.

```text
GET /style.css
GET /app.js
GET /logo.png
```

서버는 각각의 파일을 응답으로 보내고, 브라우저는 그것들을 조합해서 최종 화면을 만듭니다.

```text
HTML → 구조
CSS → 디자인
JavaScript → 동작
이미지 → 시각 자료
```

---

## 12. HTTP 메서드

HTTP 요청에는 여러 종류의 메서드가 있습니다.


| 메서드    | 의미          |
| ------ | ----------- |
| GET    | 데이터를 요청     |
| POST   | 데이터를 서버에 전송 |
| PUT    | 데이터를 전체 수정  |
| PATCH  | 데이터를 일부 수정  |
| DELETE | 데이터를 삭제     |


예시는 다음과 같습니다.

```text
게시글 보기     → GET
로그인 요청     → POST
회원 정보 수정  → PATCH 또는 PUT
게시글 삭제     → DELETE
```

---

## 13. HTTP 상태 코드

서버는 응답을 보낼 때 상태 코드를 함께 보냅니다.

상태 코드는 요청 결과를 숫자로 알려주는 값입니다.


| 상태 코드     | 의미         |
| --------- | ---------- |
| 200 OK    | 요청 성공      |
| 301 / 302 | 다른 주소로 이동  |
| 400       | 잘못된 요청     |
| 403       | 권한 없음      |
| 404       | 요청한 페이지 없음 |
| 500       | 서버 내부 오류   |


예를 들어 `404 Not Found`는 서버는 존재하지만 요청한 파일이나 페이지를 찾지 못했다는 뜻입니다.

---

## 14. localhost에서의 서버와 브라우저

직접 서버를 만들고 실행하면 보통 내 컴퓨터 안에서 테스트합니다.

예를 들어 서버 프로그램을 실행합니다.

```bash
./webserv
```

서버가 8080 포트에서 기다린다고 하면 브라우저에서 다음 주소로 접속합니다.

```text
http://localhost:8080
```

이때 통신 구조는 다음과 같습니다.

```text
브라우저 → 내 컴퓨터의 8080 포트 → webserv 서버
```

즉, 인터넷을 거치지 않고 내 컴퓨터 안에서 브라우저와 서버가 통신합니다.

`localhost`는 내 컴퓨터 자신을 의미합니다.

```text
localhost = 127.0.0.1 = 내 컴퓨터
```

## 15. 42 webserv 과제 관점

42 `webserv` 과제에서 직접 만드는 서버는 브라우저의 HTTP 요청을 받아서 HTTP 응답을 만들어 보내야 합니다.

서버가 해야 하는 일은 다음과 같습니다.

```text
1. socket()으로 통신용 문을 만든다.
2. bind()로 IP와 포트에 연결한다.
3. listen()으로 요청을 기다린다.
4. accept()로 브라우저의 연결을 받는다.
5. recv()로 HTTP 요청을 읽는다.
6. 요청 내용을 파싱한다.
7. 요청에 맞는 응답을 만든다.
8. send()로 HTTP 응답을 보낸다.
9. 연결을 정리한다.
```

흐름을 코드 관점에서 보면 다음과 같습니다.

```text
socket()
   ↓
bind()
   ↓
listen()
   ↓
accept()
   ↓
recv()
   ↓
parse HTTP request
   ↓
make HTTP response
   ↓
send()
   ↓
close()
```

---

## 16. 전체 흐름 최종 정리

브라우저와 서버의 전체 통신 흐름은 다음과 같습니다.

```text
1. 사용자가 브라우저에 주소를 입력한다.
2. 브라우저가 도메인을 IP 주소로 변환한다.
3. 브라우저가 서버의 IP와 포트로 TCP 연결을 시도한다.
4. HTTPS라면 TLS 암호화 연결을 만든다.
5. 브라우저가 HTTP 요청을 보낸다.
6. 서버가 요청을 읽고 처리한다.
7. 서버가 HTTP 응답을 보낸다.
8. 브라우저가 응답을 해석한다.
9. HTML, CSS, JavaScript, 이미지를 조합해서 화면을 보여준다.
```

---

## 17. 한 줄 요약

```text
브라우저는 요청을 보내는 프로그램이고,
서버는 특정 IP와 포트에서 기다리다가 요청을 받아 응답하는 프로그램이다.
HTTP는 이 둘이 웹에서 대화하기 위한 규칙이다.
```
